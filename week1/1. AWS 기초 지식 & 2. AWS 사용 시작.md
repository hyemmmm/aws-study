## 1. Amazon Web Service 기초 지식

### 1-1. AWS를 시작하기 위한 첫 걸음

* **대표적인 클라우드 서비스, Amazon Web Services**
    * AWS란 인터넷 쇼핑몰 Amazon.com에서 운영하는 클라우드 서비스
    * 클라우드 서비스란?
        * 어떠한 시스템을 구축하기 위해서는 컴퓨터와 데이터베이스 같은 기능이 필요한데 이런 기능을 인터넷을 통해 이용할 수 있게 해주는 서비스
    * AWS는 클라우드 서비스 중 하나로 컴퓨팅, 데이터베이스, 스토리지, 모바일, IoT, 기계 학습 등 다양한 서비스를 제공
* **점유율이 높은 AWS**
    * AWS처럼 누구나 사용할 수 있는 클라우드를 개방형 클라우드(Public Cloud)라고 함
    * AWS 외의 개방형 클라우드로는 GCP(Google Cloud Platform), Microsoft Azure 등이 있음
    * 그 중 AWS가 점유율이 가장 높음
    * AWS 고객 성공 사례: [https://aws.amazon.com/ko/solutions/case-studies/](https://aws.amazon.com/ko/solutions/case-studies/)
    * AWS 한국 사용자 모임: [https://www.awskr.org/](https://www.awskr.org/)

### 1-2. 클라우드 용어 알아두기

* **온프레미스란**
    * 온프레미스(on-premise)란 사용자가 관리하는 시설 내에 서버 등의 기기를 설치해 운용하는 환경, 이런 기기를 설치하고 운용하는 시설을 **데이터 센터**라고 한다.
    * **장점**
        * 기기를 사용자가 자유롭게 이용할 수 있고 이용 형태에 맞게 자유롭게 구성할 수 있다
        * 초기 투자 비용은 많이 들지만, 이후 운용 비용의 비율은 비교적 낮다
    * **단점**
        * 초기 투자 비용이 크고 설치나 설정에도 시간이 걸려 준비 기간이 길다
        * 기기 고장 등 예측하지 못한 상황으로 인한 추가 비용 발생할 수 있음
* **클라우드란**
    * 클라우드란, 클라우드 서비스 제공자가 서버 등의 기기를 준비하고 거기에 구축된 가상 서버나 응용 프로그램 등을 사용자에게 제공하여 이용료를 받는 형태
    * 정식 명칭은 **클라우드 컴퓨팅**
    * **장점**
        * 모든 하드웨어를 관리해주기 때문에 사용자는 하드웨어 장애에 대해 신경 쓰지 않아도 됨
    * **단점**
        * 제공되는 서비스 범위 내에서만 시스템을 이용할 수 있기 때문에 시스템 구성 자유도 낮음
        * 매월 서비스 사용료를 지불하는 형태라 전체 비용이 커질 수 있음
* **가상화**
    * 클라우드 서비스에서 사용자가 원하는 사양으로 서버를 선택하면 가상화 기술을 이용해 서버를 가상으로 생성해 사용하게 됨
    * 가상 서버는 하나 또는 클러스터링된 물리 서버 내에 만들어짐
    * 가상 서버는 물리 서버의 컴퓨터 자원을 일부 독점해 물리적으로 독립된 서버와 같이 동작
    * 스토리지나 네트워크 장비 등도 서버처럼 가상화 기술로 여러 기기처럼 취급할 수 있음 (가상 스토리지, 가상 OO ↔ 물리 OO)
* **서버리스**
    * 서비스가 이용될 때만 서버를 가동하는 방식을 가리킴
    * 서버를 사용하는 시간만큼 요금이 발생하므로 비용을 낮출 수 있음
* **개방형 클라우드와 폐쇄형 클라우드**
    * **개방형 클라우드**
        * 모든 사람이 쓸 수 있게끔 공개된 형태 (AWS가 여기에 해당)
        * 이용이 끝나면 대여해줬던 서버를 다른 사용자에게 다시 대여할 수 있음
    * **폐쇄형 클라우드**
        * 외부에 공개하지 않도록 기업 내부에 구축하는 형태
        * 온프레미스에 가까우며, 시설 내부에 클라우드 컨테이너를 두고 전용 클라우드 환경을 구축해 사용
        * 하지만 개방형 클라우드에서도 보안을 충분히 확보할 수 있고 초기 투자 비용이 커서 현재는 개방형 클라우드를 선택하는 경우가 많음
    * **혼합형 클라우드**: 계방형과 폐쇄형을 섞은 형태
* **서비스 제공 형태에 따른 클라우드 분류**
    * **SaaS**: 응용 프로그램을 서비스로 제공하는 형태
    * **PaaS**
        * 응용 프로그램을 개발하는 사용자를 위한 서비스
        * 클라우드 서비스 제공자가 OS 및 미들웨어까지 관리
        * AWS에서 관리형 서비스로 제공하는 RDS, DynamoDB, Lambda 등이 해당
    * **IaaS**
        * 응용 프로그램을 개발하는 사용자를 위한 서비스
        * 클라우드 서비스 제공자가 서버 및 네트워크 기능만 제공
        * AWS의 EC2, VPC, EBS 등이 해당

### 1-3 AWS를 이해하기 위한 6가지 특징

* **AWS와 사용자 간의 책임 공유**
    * AWS가 책임지는 범위를 ‘공동 책임 모델’이라는 형태로 정의
    * 책임 범위는 이용하는 서비스에 따라 다름
* **글로벌 시스템 구축 가능**
    * AWS 데이터 센터는 전 세계에 지역별로 리전이라는 단위로 분리되어 있음 (2022년 기준 26개)
    * 각 리전에는 가용 영역(Available Zone, 이후 AZ)이 여러 개 존재하고, 하나 이상의 데이터 센터로 구성됨
    * 각 AZ는 서로 다른 영역에 있기 때문에 장애가 발생하더라도 다른 AZ에서 서비스 제공
    * 이용할 리전을 선택해 글로벌 서비스 구축
* **사용한 만큼만 이용료 지불**
    * AWS는 종량 과금제
    * 대부분 시간당 요금이 부과되도록 설정(요금X이용시간)
    * 요금은 사용하는 인스턴스의 종류와 스토리지에 따라 증가
* **서버의 자원과 수를 설정에서 쉽게 변경**
    * 구축된 서비스의 사용자가 증가했을 때 서버 수를 늘리거나 사양을 높이는 등 유연하게 대응 가능
* **장애를 예상한 설계**
    * AWS는 장애는 언제라도 발생할 수 있다는 전제로 설계, 이를 **Design for Failure**라고 함
    * 시스템을 중단 없이 계속해서 실행할 수 있는 능력을 가용성이라고 하고 이를 제공해줌
    * 스토리지 서비스인 S3처럼 자동으로 여러 AZ에 배치하거나 데이터 영역을 확장하는 서비스도 있음
* **설계의 모범이 되는 프레임워크**
    * AWS는 **Well-Architected** 라는 아키텍처 모범 사례를 모아 사용자가 안전하고 효율적인 인프라를 구축할 수 있게 지원하는 프레임워크를 제공
    * **Well-Architected 프레임워크**의 6가지 원칙: 운영 우수성, 보안, 안정성, 성능 효율성, 비용 최적화, 지속 가능성
    * [https://wa.aws.amazon.com/index.ko.html](https://wa.aws.amazon.com/index.ko.html)
    * 초보자가 보기는 어렵고 고도화된 시스템을 만들 때 참고 가능

### 1-4. 서비스 분류 및 대표적인 서비스

* **AWS 서비스 분류**
    * 서비스 특성별로 분류해 이해하면 AWS 서비스를 파악하기 편함
    * 이 책의 카테고리처럼 컴퓨팅, 스토리지, 네트워크 및 콘텐츠 전송, 데이터베이스, 보안 & 자격증명, 기타 서비스(데이터 분석, 기계 학습 등)로 분류할 수 있음
* **대표적인 서비스**
    * 컴퓨팅: 가상 서버인 **EC2**, 컨테이너 서비스 **ECS**
    * 스토리지: **EBS**, **S3**
    * 데이터베이스: **RDS**, **DynamoDB**
    * 네트워크: 가상의 폐쇄형 클라우드 환경을 구현하는 **VPC**, 고기능 DNS인 **Route53**, 서비스 처리를 분산하는 **ELB**
    * **Lightsail**이나 **Beanstalk**처럼 자주 쓰이는 서비스를 조합해 패키지로 구성한 서비스도 있음
* **관리형 서비스, 비관리형 서비스**
    * 관리형 서비스란, 서비스를 이용하기 위한 관리나 운영을 클라우드 서비스 제공자가 수행하는 것. 대부분의 서비스가 관리형 서비스에 해당(RDS, S3 등)
    * 완전관리형 서비스: 사용자가 직접 관리할 필요가 없는 서비스(데이터양에 따라 저장 영역을 자동으로 확장하는 Aurora)
    * 비관리형 서비스: 사용자가 직접 OS 설정을 관리하거나 장애 대응을 해야 하는 서비스(EC2)

### 1-5. AWS 도입 사례

* **AWS가 공개한 사례**: AWS는 많은 도입 사례를 찾아볼 수 있고 공식 사이트에서도 사례를 볼 수 있음
* **게임 업계 사례**
    * 자원의 유연성과 글로벌 인프라를 제공하는 특징을 활용함
    * 예) 게임 내에서 특정 이벤트를 개최할때 평소보다 많은 사용자 접속이 되어도 일시적으로 컴퓨팅 자원을 늘리고 접속이 줄어들면 자원을 반환하는 형태로 대응(자원의 유연성)
    * 예) 세계 각지의 사용자에게 빠르게 데이터를 전달하기 위해 콘텐츠 전달 서비스 활용(글로벌 인프라)
* **헬스케어 업계 사례**
    * 요구 조건: 빅데이터를 활용하는 경우가 많아 방대한 데이터를 빠르게 분석하고 처리하는 능력 필요, 민감 정보를 다루므로 보안 요구 조건 높음
    * 해결: 데이터 분석에 유용한 고성능 CPU 및 GPU를 탑재한 컴퓨팅 서비스와 방대한 데이터를 보관하는 스토리지 서비스 제공, VPC나 AWS WAF를 이용해 원하는 수준의 보안 시스템 구축
* **제조 업계 사례**: 온프레미스 → AWS 마이그레이션, IoT를 활용한 서비스 구축 등 다양한 사례가 있음
* **금융 업계 사례**: AWS는 PCI-DSS(Payment Card Industry Data Security Standard) 같은 최신 보안 표준의 대응도 가능하여 금융 업계에서도 사용하고 있음

## 2. Amazon Web Service 사용 시작

### 2-1. 계정을 만들고 AWS 이용 시작

* **AWS 계정 생성**: 계정을 만들 때 AWS 지원 플랜을 선택(Basic, Developer, Business, Enterprise)
* **AWS 계정 로그인**
    * **루트 사용자**: 모든 권한을 가진 사용자(AWS 계정 자체의 구성 변경, 계약 해지, 계약 변경 등)
    * **IAM 사용자**: 보안을 위해 IAM이라는 서비스를 이용해 별도의 IAM 계정을 만들고 적절한 권한을 부여해 서비스 사용
    * 처음 로그인 후 IAM 계정 생성 후 IAM 계정만 사용하는 것이 권장됨

### 2-2. AWS 서비스를 사용하는 세 가지 방법

* **웹 브라우저에서 AWS 조작**
* **명령줄 도구로 AWS 조작**
    * AWS Command Line Interface(AWS CLI) 설치 후 CLI 명령을 사용해 AWS 모든 자원의 기능을 조작할 수 있음
    * 예) 보안 그룹을 만드는 CLI 명령
    ```bash
    $ aws ec2 create-security-group --group-name sample-sg --description "Sample sg"
    ```
    * 공식 문서 - [https://docs.aws.amazon.com/cli/latest/reference/](https://docs.aws.amazon.com/cli/latest/reference/)
    * 웹 브라우저에서는 UI 변경시 조작 실수나 어려움이 발생할 수 있는 것에 비해, CLI는 자주 사용하는 명령을 저장해두고 사용할 수 있으므로 조작을 자동화하여 실수를 줄일 수 있어, **재현성이 높다**는 장점이 있음
* **프로그램 방식으로 AWS 조작**
    * 프로그램을 이용해 조작하기 위해서는 AWS SDK라는 개발 키트를 설치해야함
    * 8가지 개발 언어 지원(JavaScript, Python, PHP, .NET, Ruby, Java, Go, C++)
    * React, Angular, 안드로이드, iOS, IoT 기기용 SDK도 제공
    * AWS 자원 조작 뿐 아니라 응용 프로그램의 일부로 AWS 자원을 조합하거나 S3에 파일을 저장하는 등의 다양한 기능 제공

### 2-3. AWS 서비스 이용료 시각화

* **AWS 비용 관리**
    * 이용한 만큼 매월 이용료가 청구되는 종량 과금제
    * 큰 시스템일 수록 많은 AWS 서비스를 이용하므로 요금 관리에 어려움이 있음
    * 이러한 문제를 예방하기 위한 방법이 **AWS Billing and Cost Management**
* **AWS Billing and Cost Management란**
    * 웹 참고 (우측 사용자 계정 클릭 - ‘결제 및 비용 관리’ 클릭)
    * 청구서, Cost & Usage reports, Cost explorer, Budgets 등 관리
* **비용 관리 정책**
    * **Cost Explorer**: 비용 상황 파악, 어디에서 비용이 발생하는지 확인하고 비용 최적화 실시 가능
    * **Budgets**: 예산을 설정하면 금액을 초과할 때 알림을 보내줌

### 2-4. AWS 시스템이 구축되는 위치 선택

* **리전은 세계에 존재하는 지역**
    * 리전: 전 세계 각 지역에 지리적으로 떨어진 영역이 존재, 각 리전은 AWS의 사설 네트워크로 연결되어 있음
    * 세계 각지에 리전이 존재했을 때의 장점
        * 글로벌 시스템을 구축할 수 있어 사용자의 통신 지연 줄일 수 있음
        * 법적 요구 사항으로 특정 국가에 시스템 구축 필요시 특정 리전을 사용해 요건 충족 가능
        * 재난으로 인해 특정 리전에서 시스템 사용불가여도 다른 리전에서 시스템 가동 가능
    * 웹 브라우저에서 리전별로 분리해 자원 관리 가능
* **가용성 영역은 리전 내의 독립적인 위치**
    * 각 리전에는 복수의 가용 영역(AZ)가 존재하고 모두 독립적
    * AZ에는 AZ 이름(ap-northeast-2a)과 AZ ID(apne2-az1)가 있고, AWS 계정마다 AZ 이름에 대응하는 AZ ID는 다름
    * AZ ID가 동일하면 지리적으로 같은 장소, 장애가 발생하면 AZ ID를 확인하기
    * 서비스 생성시 여러 AZ를 지정해 자원을 생성하면 시스템 가용성을 높일 수 있음
    * S3의 경우에는 AWS에서 자동으로 3개 이상의 AZ를 사용하게 함
