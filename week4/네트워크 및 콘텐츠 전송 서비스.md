## 1. 네트워크 중요 용어 익히기

### 1) IP 주소
- **IP 주소란?** 네트워크상에서 장치(호스트)를 논리적으로 식별하기 위해 부여하는 고유한 주소
- **IPv4(Internet Protocol version 4)**: 현재 가장 보편적으로 사용되는 32비트 기반의 네트워크 계층 프로토콜
  - 4개의 숫자(각 숫자는 0~255의 범위를 가짐)를 점으로 구분해 표시
  - IPv6: IPv4 주소 고갈 문제로 도입되었으며, 모바일 및 클라우드 환경을 중심으로 도입률이 꾸준히 증가하고 있음
- **IP 주소 구분**
  - **퍼블릭 IP 주소**: 전 세계에서 특정할 수 있는 주소, 글로벌 IP 주소라고도 함 
  - **프라이빗 IP 주소**: 닫힌 네트워크(근거리 통신-LAN) 내에서만 식별할 수 있는 IP 주소, 인터넷 직접 연결 불가능
    - 10.0.0.0/8: 10.0.0.0 ~ 10.255.255.255
    - 172.16.0.0/12: 172.16.0.0 ~ 172.31.255.255
    - 192.168.0.0/16: 192.168.0.0 ~ 192.168.255.255
- **CIDR 블록**: IP 주소를 이용해 네트워크 범위를 정의하는 것
  - IP 주소는 네트워크 부분과 호스트(네트워크 안에서 접속할 수 있는 서버 등) 부분으로 나눌 수 있음
    - **서브넷 마스크**: IP 주소에서 네트워크 부분(Network ID)과 호스트 부분(Host ID)을 구분하기 위해 사용하는 비트 값
      - ex. '172.31.0.0/16'
        - '/16': 서브넷 마스크, 앞의 16비트(2옥텟)가 네트워크 부분임을 의미, 2진수 마스크로 표기했을 때 왼쪽부터 채워진 '1'의 개수
        - '172. 31': 네트워크 부분, 고정되어 변하지 않는 범위
        - '0.0': 호스트 부분, 장치에 할당 가능한 가변 범위
    - **네트워크 주소**: 호스트 부분이 모두 0인 주소, 할당 불가능
    - **브로드캐스트 주소**: 호스트 부분이 모두 1인 주소, 할당 불가능

### 2) 방화벽
- **방화벽이란?** 미리 정의된 보안 규칙에 따라 트래픽을 모니터링하고 제어(허용/차단)하는 보안 시스템, 보안 위협으로부터 시스템을 지키는 역할을 함
  - ex. AWS의 보안 그룹, 네트워크 ACL

### 3) 부하 분산
- **로드 밸런서(Load Balancer)**: 부하를 여러 서버에 분산시키기 위해 사용하는 장치 

### 4) 라우팅
- **라우팅(routing)이란?**: 라우터가 IP 주소까지의 최적의 경로를 찾아서 결정하는 것
- **라우팅 테이블**: 각 라우터가 목적지 IP 주소를 향해 이동해야 하는 네트워크를 결정하는데 기반이 되는 경로 정보
  - 인터넷은 여러 네트워크의 집합체이고, 각 네트워크의 라우터가 IP 주소로 향하기 위한 최적의 경로를 선택해 줌
  - ex. Amazon VPC의 라우팅 테이블 기능 

### 5) DNS
- **도메인이란?**: 웹 사이트에 접속하기 위해 IP 주소를 문자 형태로 바꾼 주소, URL의 한 구성 요소를 이룸 (ex. xxx.com)
- **DNS(Domain Name System)**: 도메인 이름을 연결된 실제 IP 주소로 변환해주는 시스템
  - 이러한 기능을 '이름 해석'이라고 함

## 2. Amazon VPC

### 1) VPC란
- AWS에서 생성할 수 있는 프라이빗 가상 네트워크 공간
  - VPC에 EC2와 같은 AWS 자원을 배치해 이용
- **VPC 생성**
  - CIDR 블록(IP 주소 범위)을 지정
    - CIDR /16 (10.0.0.0/16)
      - $2^{16}$ = 65,536개의 IP 주소 사용 가능 (실제로는 65,531개 사용 가능)
      - 일반적인 운영 환경에서 사용
    - CIDR /28 (10.0.0.0/28)
      - $2^{4}$ = 16개의 IP 주소 사용 가능 (실제로는 11개 사용 가능)
      - 특정 목적의 아주 작은 네트워크를 위해 사용
  - 일반적으로 프라이빗 IP 주소 사용
    - 외부 퍼블릭 IP와 겹치지 않기 위해
    - 외부 네트워크와의 접속을 검토하고 있다면 접속할 네트워크의 CIDR 블록과 중복되지 않도록 지정
  - 자원 증가를 대비하여 호스트 주소를 충분히 확보하는 것이 필요

### 2) VPC 및 서브넷 생성
- 관리 콘솔에서 생성하는 경우, 다음 항목을 설정해야 함
  - VPC 이름: Name 태그로 저장됨, 자원 식별을 위해 사용됨
  - CIDR 블록: VPC에서 사용할 IPv4 주소 범위
  - IPv6 설정: 필요에 따라 설정
  - 테넌시 (전용 하드웨어 사용 여부): 필요에 따라 설정
- **서브넷**: VPC 안에 존재하는 더 작은 네트워크 단위, EC2와 같은 자원이 배치되는 실질적인 장소
  - 하나의 서브넷은 하나의 AZ에 속해야 함, 여러 AZ에 자원을 배치하려면 AZ별로 서브넷도 각각 여러 개 만들어야 함
  - 서브넷의 CIDR 블록은 VPC CIDR 블록의 하위 범위로 지정되어야 함
    - ex. VPC가 /16이면 서브넷은 /16~/28 사이에서 생성 가능

### 3) VPC의 주요 기능
**① 라우팅 테이블(네트워크의 경로 정보)**
- VPC에 라우팅 테이블을 생성, 생성된 라우팅 테이블을 각 서브넷과 연결함
  - cf. 온프레미스 환경: 라우터와 같은 네트워크 기기에 라우팅 테이블 설정
  - VPC를 만들 때 기본적으로 하나의 라우팅 테이블이 만들어짐
    - 기본 상태에서는 VPC 내의 라우팅 정보만 있으므로, 외부와 통신하려면 외부 라우팅 정보를 추가해야 함
    - ex. 0.0.0.0/0(모든 네트워크) 경로를 인터넷 게이트웨이로 등록하면 인터넷과 통신할 수 있는 퍼블릭 서브넷이 됨

**② 인터넷 게이트웨이**
- 서브넷 안에 있는 자원이 인터넷과 통신할 수 있게 하기 위한 기능(관문)
- **퍼블릭 서브넷**: 인터넷 게이트웨이로 가는 경로가 설정된 서브넷
  - 외부 인터넷과 통신하려면 EC2에 퍼블릭 IP 또는 Elastic IP가 할당되어야 함

    | 구분 | 퍼블릭 IP | Elastic IP |
    | :--- | :--- | :--- | 
    | **특징** | 인스턴스 시작 시 자동 부여 | 사용자가 직접 할당받아 연결하는 고정 IP | 
    | **지속성** | 중지 후 시작 시 변경됨 (재부팅 시에는 유지) | 인스턴스를 삭제하기 전까지 영구 고정 |
    | **요금** | 시간당 비용 발생 (2024년부터 정책 변경) | 시간당 비용 발생 (미사용 시에도 부과) |
    | **용도** | 임시적인 인터넷 통신 | 방화벽 등에서 IP 주소를 고정해야할 때 |

- **프라이빗 서브넷**: 인터넷 게이트웨이로 향하는 경로가 없어 인터넷과 통신할 수 없는 서브넷
  - cf. 프라이빗 서브넷의 자원이 패치 업데이트 등을 위해 나가는 통신(Outbound)만 필요하다면 인터넷 게이트웨이 대신 NAT 게이트웨이 사용

**③ NAT 게이트웨이**
- NAT(Network Address Translation): 프라이빗 IP 주소를 퍼블릭 IP 주소로 변환하는 것
- 프라이빗 서브넷의 IP 주소를 NAT 게이트웨이의 퍼블릭 IP로 변환해 인터넷과 통신할 수 있게 해줌
- VPC 내에서 인터넷과 통신해야 하는 경우(ex. 인터넷 상 소프트웨어 패키지 다운로드) 사용
  - 인터넷에서 VPC로의 통신은 불가능한 단방향 통신으로 AWS 외부와 안전하게 통신 가능

**④ VPC 접근 제어 및 통신 로그 확인**
- 네트워크 접근 제어 목록(네트워크 ACL): 서브넷 단위로 접근 제어 설정 가능
- 네트워크 ACL vs. 보안 그룹

  | 구분 | 네트워크 ACL | 보안 그룹 |
  | :--- | :--- | :--- | 
  | **설정 단위** | 서브넷 단위로 설정 | 인스턴스(자원) 단위로 설정 |
  | **허용/거부 설정** | 허용 및 거부 설정 가능 | 허용만 설정 가능 (허용하지 않은 모든 것은 거부) |
  | **상태 저장 여부** | 상태 비저장(Stateless), 단일 패킷만 확인하므로 정보를 저장하지 않음 → 인바운드와 아웃바운드 규칙을 모두 명시적으로 만들어야 통신 가능 | 상태 저장(Stateful), 패킷과 관련된 세션까지 확인하므로 정보를 저장 → 한쪽(예: 인바운드)이 허용되면 반대쪽(예: 아웃바운드) 응답은 자동 허용 |
  | **규칙 우선 순위** | 등록된 규칙의 번호순으로 트래픽 허용 및 거부(거부 규칙을 최우선 번호로 배치하여 보안  강화) | 등록된 모든 규칙을 평가해 트래픽 허용 |
  | **실무적 특징** | 블랙리스트(Blacklist) 방식: 특정 나쁜 IP나 대역을 입구에서 차단할 때 유리함 | 화이트리스트(Whitelist) 방식: 허용된 안전한 요청만 통과시킬 때 사용함 |
  | **주요 용도** | 특정 국가/IP 차단, 서브넷 간의 큰 흐름 제어 | 웹 서버, DB 서버 등 역할별 접근 제어 |

- 서브넷 전체에서 네트워크 접근을 허용하거나 거부하려는 경우 보안 그룹에서 추가 설정을 해 안전성을 높일 수 있음
- **VPC 흐름 로그**: VPC 내의 네트워크 인터페이스(ENI)에서 송수신되는 IP 트래픽 정보를 저장하는 기능, 네트워크 ACL이나 보안 그룹을 통해 허용되거나 거부된 통신 상황 확인 가능 (CloudWatch Logs, S3에 저장 가능)  
  - CloudWatch Logs: 지표 필터와 경보(Alarm)를 연동해 실시간 메일 알림 가능
  - S3: 요금이 더 저렴함
- **네트워크 인터페이스(Elastic Network Interface)**: VPC 내부 자원에 부착된 가상 네트워크 인터페이스
  - 네트워크 인터페이스(ENI)에 서브넷의 IP 주소가 할당됨
  - VPC 흐름 로그는 이 네트워크 인터페이스 단위를 기준으로 출력됨
  - 도착지/출발지의 IP 주소와 포트, 전송 허용/거부 여부 등의 정보가 포함됨
  - VPC 내부에서 통신하는 모든 서비스(ELB, RDS, Redshift)는 각각의 ENI를 통해 로그를 생성함

### 4) VPC와의 연결
**① VPC와 외부 네트워크 연결**
- **VPC 피어링**
  - 서로 다른 두 개의 VPC를 연결해 통신
  - 두 개의 VPC 간에 수행되기 때문에, 세 개의 VPC가 통신하는 경우에는 각 VPC끼리 따로 피어링을 구성해야 함
  - 연결된 VPC의 CIDR 블록은 겹치면 안 됨
- **AWS Transit Gateway**
  - VPC 연결을 하나의 중앙 허브에서 관리
  - VPN과 같이 온프레미스 환경과의 연결에도 사용 가능
  - 요금은 VPC 피어링을 사용하는 것보다 높으나, 많은 VPC를 사용할 시 추천 

**② VPC와 VPC 외 AWS 서비스를 연결**
- **VPC 엔드포인트**
  - 인터넷을 거치지 않고 프라이빗 네트워크로 S3와 같은 AWS 서비스에 연결
  - 게이트웨이 엔드포인트: S3, DynamoDB에서 사용하는 VPC 엔드포인트
- **인터페이스 엔드포인트**
  - AWS PrivateLink라는 기능을 사용해 서브넷에 서비스 접속용 ENI(네트워크 인터페이스)를 생성, 해당 ENI에 할당된 프라이빗 IP로 통신
    - AWS PrivateLink: 트래픽을 공용 인터넷에 노출시키지 않고 VPC, AWS 서비스, 온프레미스 간에 전용 연결을 제공하는 기술
  - S3, DynamoDB만을 지원하는 게이트웨이 엔드포인트와 달리 많은 AWS 서비스를 지원함

**③ VPC와 온프레미스 네트워크 연결**
- **AWS Site-to-Site VPN**
  - 온프레미스 환경의 네트워크와 VPC를 VPN으로 연결
    - VPN(Virtual Private Network): 가상의 프라이빗 통로를 구축하여 외부 네트워크와 프라이빗 IP 주소로 통신 가능
    - 고객 게이트웨이(온프레미스 쪽 라우터 기기)와 가상 프라이빗 게이트웨이(VPC 쪽 VPN 종단점)를 통해 프라이빗 접속이 이루어짐
      - IPsec: IP 통신 단계에서 데이터를 암호화하고 상대방을 인증하기 위한 프로토콜 집합 → 이 기술을 활용하여 VPN를 생성함
    - VPN 터널의 이중화 확보: 물리적 장애에 대비해 자동으로 서로 다른 AZ에 위치한 2개의 터널 생성
- **AWS Client VPN**
  - 온프레미스 환경의 클라이언트 단말(PC)과 VPC를 VPN으로 연결
    - 온프레미스 환경 전체가 아니라 특정 단말과 비공개로 연결하고 싶을 때 사용
- **AWS Direct Connect**
  - 온프레미스 환경과 AWS(VPC 외 AWS 서비스 포함)를 전용선으로 연결
    - 온프레미스 네트워크는 AWS 데이터 센터에 직접 연결되는 것이 아니라 Direct Connect 로케이션이라는 중간 연결 지점을 통해 연결
  - VPN 연결과 비교해 빠르고 안정적인 네트워크로 연결 가능
  - 이용 비용이 높고, 서비스 이용 신청 후 이용할 수 있을 때까지의 준비 기간이 긺

### 5) VPC 구성 예
**실행 위치에 따른 AWS 서비스의 유형**
- **글로벌 서비스**: 특정 리전에 의존하지 않고 실행되는 서비스
  - ex. CloudFront, IAM 등
- **리전 서비스**: 리전 내, VPC 외부에서 실행되는 서비스
  - ex. S3, DynamoDB 등
- **AZ 서비스**: VPC 내에서 실행되는 서비스
  - ex. EC2, RDS 등

**① WEB + DB + Amazon CloudFront**
- 온프레미스 환경의 웹 서버, DB 서버 환경을 AWS로 마이그레이션할 때 자주 볼 수 있는 구성
  - 하나의 리전에 프라이빗 네트워크로 VPC를 생성한 다음, 웹 서버(EC2) 및 DB 서버(RDS)와 같은 자원을 배치
  - 서버 역할별로 서브넷을 생성, 멀티 AZ 대응을 위해 각 역할별로 2개의 서브넷을 준비하여 다른 AZ에 배치
  - ALB(Application Load Balancer)만 퍼블릭 서브넷에 배치하여 VPC 외부로부터 통신 수신
  - CloudFront를 VPC 외부에 배치해서 웹 콘텐츠를 캐싱 → 지연 시간을 최소화하여 콘텐츠에 빠르게 접근할 수 있게 함

**② VPC 엔드포인트를 사용하는 S3 연결**
- 구성 예시 ①에 VPC 엔드포인트와 S3 버킷을 추가한 구성
  - 데이터베이스(RDS)에서 취득한 데이터를 S3에 저장하거나 S3에서 얻은 데이터를 다른 데이터베이스로 가져오는 프로세스를 가정
  - VPC 엔드포인트를 경유하므로 인터넷에 통신 데이터가 나오지 않음

**③ 온프레미스 환경에서 프라이빗 연결**
- 온프레미스 환경 전용 프라이빗 네트워크: Direct Connect와 Site-to-Site VPN을 모두 사용해 온프레미스 환경과 프라이빗 연결을 하는 구성
  - VPC와 온프레미스 환경만 통신 가능, 인터넷과는 통신하지 않음
  - 프라이빗 서브넷만 해당(ALB도 프라이빗 네트워크에서 접속 가능)
  - Direct Connect와 Site-to-Site VPN로 연결을 중복 → 장애 시 전환 가능

## 3. ELB(Elastic Load Balancing)
### 1) ELB란
- AWS에서 제공하는 로드 밸런서 서비스

### 2) 로드 밸런서의 주요 기능
- 트래픽(네트워크를 통해 흐르는 데이터)을 로드 밸런싱
- 트래픽을 보내는 서버(EC2와 같은 인스턴스)가 멈추지 않고 실행적인지 정기적으로 확인하여 가용성 체크

### 3) ELB가 제공하는 기본 기능
- **부하 분산**
  - 트래픽을 여러 대상으로 전달하도록 지정하여 사용자의 접속 분산
  - 각 대상을 다른 AZ에 배치하여 가용성 향상 
  - 부하 상황에 따라 자동으로 스케일됨(유입되는 트래픽 양에 따라 로드 밸런서의 처리 용량을 자동으로 조절) → 이때 IP 주소가 변경될 수 있으므로 DNS 주소로 접속해야 함
- **대상 모니터링**
  - 대상에 대한 연결과 상태 감시 (CloudWatch 지표 취득 가능)
  - 비정상적인 동작 감지 시, 대상을 자동으로 분리함(문제가 발생한 서버로의 전달 중단)
- **보안 기능**
  - 보안 그룹을 비롯한 AWS의 기본적인 보안 서비스 적용 가능
    - NLB(Network Load Balancer)는 보안 그룹 설정 불가능
  - SSL/TLS 서버 인증서를 설정해 암호화 통신 수행 가능

### 4) ELB 유형

| 이름 | 설명 |
| :--- | :--- |
| **Application Load Balancer(ALB)** | - HTTP 트래픽과 HTTPS 트래픽의 부하 분산 가능 <br> - 레이어 7(응용 프로그램 계층)에서 동작: HTTP 헤더, URL 경로(/api, /blog), 호스트 이름 기반의 정교한 콘텐츠 기반 라우팅 지원 <br> - 마이크로 서비스 및 컨테이너 최적화: 하나의 로드 밸런서로 여러 독립적인 마이크로서비스를 통합 관리할 수 있으며, 동적으로 변하는 컨테이너의 포트를 자동으로 감지하여 트래픽을 전달 |
| **Network Load Balancer(NLB)** | - 레이어 4(전송 계층)에서 동작: HTTP/HTTPS 외에 TCP, UDP, TLS 트래픽의 부하 분산도 가능 <br> - 수백만 건 이상 요청이 발생하는 대규모 트래픽에서도 속도가 빠름
| **Classic Load Balancer(CLB)** | - ALB, NLB 서비스 이전부터 제공된 구형 로드 밸런서 <br>  - 레이어 4(전송 계층) 및 레이어 7(응용 프로그램)에서 동작 <br>  - 예전 아키텍처를 이용해야 하는 등 특별한 경우를 제외하고는 기본적으로 ALB, NLB 이용을 권장 |
| **Gateway Load Balancer(GWLB)** | - 타사 보안 장비(방화벽, IDS/IPS 등)를 손쉽게 사용할 수 있게 함 <br> - 레이어 3(네트워크 계층)에서 동작 <br> - 복잡한 VPC 피어링이나 라우팅 설정 없이도 GWLB 엔드포인트를 통해 트래픽을 보안 장비로 투명하게 전달 |
- ALB: 웹 사이트, 웹 시스템의 부하 분산에 사용
- NLB: ALB로 구현할 수 없는 미세한 제어나 HTTP/HTTPS 이외의 프로토콜을 이용하는 경우 사용

### 5) ELB 요금
- 1시간당 로드 밸런서 사용료(고정) + 로드 밸런서 커페서티 유닛(LCU) 이용료(변동)
- LCU: 초당 새 연결 수, 활성 연결 수, 처리된 바이트, 규칙 평가를 측정하여 가장 사용량이 많은 항목에 대해서만 요금이 부과됨, 대규모 시스템 운영 시 고려

## 4. DNS 서비스
### 1) Route 53란
- AWS가 제공하는 DNS 서비스
- 완전 관리형으로 제공되므로 온프레미스로 구축한 DNS 서버와 달리 유지 관리 및 작업이 필요 없음
- 모든 리전에서 공통으로 사용하는 글로벌 서비스
- 등록한 각 도메인에 대해 호스팅 영역이 만들어지고, 호스팅 영역별로 설정이 관리됨
- 높은 가용성, 높은 비용 대비 성능 제공

### 2) Route 53의 기능
- 도메인 등록 기능
  - 'xxx.com'과 같은 임의의 도메인 이름을 DNS 레코드로 등록 가능
    - DNS 레코드: 도메인과 그에 연결된 IP 주소 및 정보를 기록한 데이터, DNS는 도메인 주소 조회 요청이 있을 때, 해당 도메인에 매핑된 IP 주소를 찾아 반환함
  - 신규 도메인 등록 시 요금 발생, '.com', '.kr'과 같은 TLD(Top Level Domain, 최상위 도메인)의 종류에 따라 이용 요금이 달라짐

### 3) 라우팅 정책 
- 생성한 DNS 레코드별로 라우팅 정책 설정 가능
  - 여기서의 라우팅은 도메인 이름에 대한 IP 주소를 어떻게 반환하는지를 의미   

| 이름 | 설명 |
| :--- | :--- |
| **단순 라우팅** | - 하나의 도메인에 하나의 IP 주소만 연결 |
| **가중치 기반 라우팅** | - 여러 개의 라우팅 대상을 지정한 가중치(0-255) 비율에 따라 트래픽 분산 <br> - ex. X.X.X.X : 70%, Y.Y.Y.Y: 30%
| **지리적 위치 라우팅** | - 조회한 곳의 위치 정보에 따라 어떤 IP 주소를 반환할지 결정 |
| **지연 시간 라우팅** | - 항상 레이턴시(데이터 처리 지연 시간)가 최소인 자원의 IP 주소를 우선적으로 반환 |  
| **장애 조치 라우팅** | - 프라이머리 IP에 장애가 발생하면 보조로 라우팅 |
| **다중값 응답 라우팅** | - 최대 8개의 상태가 양호한 레코드를 무작위로 응답 <br> - 단순 라우팅과 달리 상태 확인 결과가 정상인 IP만 반환 |

### 4) 자원 상태 확인
- Route 53의 상태 확인 기능을 통해 대상이 정상적으로 실행 중인지 확인 가능
  - 엔드포인트 확인
    - IP 주소 또는 도메인 이름을 대상으로 연결 시도, 반환되는 응답 확인 → 요금 발생
    - 고급 설정 항목으로 요청 간격 및 상태를 확인 가능, 특정 리전만 선택도 가능 → 고급 설정 항목당 추가 요금 발생 
  - 상태 확인 계산: 최대 256개의 개별 상태 확인 결과를 논리 연산(AND, OR, NOT)으로 조합하여 하나의 상위 상태 확인 결과 도출 가능
  - CloudWatch 알람 상태 확인

## 5. CloudFront
### 1) CloudFront란
- AWS가 제공하는 콘텐츠 전달 네트워크(CDN) 서비스
  - CDN: 대용량 디지털 콘텐츠를 인터넷에서 효율적으로 사용자에게 전달하기 위한 네트워크
  - CDN은 데이터 본체를 저장하고 있는 원본 서버에서 그 데이터의 복사본(캐시)을 전 세계에 존재하는 캐시 서버에 저장 → 사용자의 위치에서 가장 가까운 곳에 있는 캐시 서버로 연결해 빠르게 데이터를 받아볼 수 있게 함 (cf. 지리적 위치 라우팅)
- 캐시 서버를 전 세계 엣지 로케이션(소규모 데이터 센터 거점)에 배치하고 AWS 리전에 기능을 제공함

### 2) CloudFront의 장점
- 대용량 콘텐츠의 빠른 배포
  - 전 세계 사용자에게 대용량 콘텐츠 전달 가능
- 보안 향상
  - 자동으로 SSL/TLS 암호화 수행
  - 자동으로 생성된 도메인 이름이 할당되며 해당 도메인 이름을 가진 SSL/TLS 인증서가 설정됨
  - AWS Shield라는 무료 DDoS 보호 서비스가 자동으로 적용됨
- 가용성 향상
  - 전 세계 엣지 로케이션에서 대상 데이터의 캐시를 전송함 → 원본 서버에 대한 부하가 분산되고, 원본 서버에서 장애가 발생했을 때의 동작도 설정 가능

### 3) CloudFront 설정
- 배포
  - 원본 서버에 배치된 파일 정보의 단위 (cf. EC2-인스턴스 단위)
  - 하나의 배포에 하나의 CloudFront 도메인이 할당되며, 사용자가 이 도메인에 접속 시 엣지 로케이션을 통해 통신하도록 함
  - cf. AWS WAF(방화벽): 사용자의 요청이 CloudFromt의 엣지 로케이션에 도달하기 전, IP 제한 및 취약점에 대한 대응 수행 가능, 유료 
- 원본 설정
  - 원본이 되는 자원(S3, ELB 등)에 대해 설정
  - 도메인 단위로 설정 가능
  - CloudFront가 원본에 접근할 때 부여하는 헤더 정보 설정 가능
- 기타 설정
  - 동작
    - 프로토콜 및 HTTP 방법 설정
    - 상세한 캐시 설정
    - 서비스에서 사용되는 경로 패턴 단위 제어 ex. '*.mp4'
  - 오류 페이지
    - 원본에서 장애가 발생했을 때 동작 설정
    - HTTP 오류 코드별 응답 정의
    - 에러가 발생했을 때 동작 설정('동작' 설정과 조합)
  - 지리적 제한
    - 특정 위치로부터의 접속을 허가 또는 차단
  - 무효화
    - 엣지 로케이션에 저장된 기존 캐시를 만료 시간(TTL)과 상관없이 수동으로 즉시 삭제

### 4) CloudFront 이용료
- 실제로 사용한 데이터 전송량, CloudFront에 대한 요청 수에 따라 요금이 결정됨
  - 데이터 전송량 월 1TB, HTTPS 요청 수 월 1,000만 건까지는 무료로 이용 가능
