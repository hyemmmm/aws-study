## 1. S3 API 더 알아보기

### 1) HEAD API 활용
- **동작:** S3의 `HEAD Object` API 호출
- **특징:** 데이터 본문(Body)은 가져오지 않고, 메타데이터(Header)만 조회
    - 파일의 존재 여부(`isFileExists`)를 효율적으로 확인할 수 있음

### 2) 멀티파트 파일 업로드 (대용량 파일 처리)
- **개념:** 하나의 대용량 파일을 여러 조각(Part)으로 나누어 병렬(Parallel)로 전송한 뒤, S3에서 합치는 기술
- **사용 기준:**
  - **필수:** 단일 PUT 최대 크기인 **5GB** 초과 시
  - **권장:** **100MB** 이상의 파일부터 권장 (안정성 및 속도 향상을 위해)
- **구현 (S3TransferManager):**
  - `S3Client`로 복잡한 과정을 직접 구현하기보다 `S3TransferManager` 사용을 권장
    - AWS SDK for Java 2.x를 위한 고수준 파일 전송 유틸리티
    - 파일 크기에 따라 일반 업로드와 멀티파트 업로드를 자동으로 판단하여 업로드를 수행함

## 2. 콘텐츠 서빙 아키텍처

### 데이터 성격별 서빙 전략
| 구분 | 추천 기술 | 특징 |
| :--- | :--- | :--- |
| **공개 데이터** (ex. 블로그 이미지, 배너) | **CloudFront URL** | **캐싱 O** (CDN을 통한 속도 향상), **보안 X** (영구적 접근 제공) |
| **비공개 데이터** (ex. 영수증, 계약서) | **S3 Presigned URL** | **캐싱 X** (S3에 직접 접근), **보안 O** (유효기간 동안만 접근 허용) |
| **유료 콘텐츠** (ex. 강의 영상) | **CloudFront Signed URL** | **캐싱 O** (CDN을 통한 속도 향상), **보안 O** (유효기간 동안만 접근 허용) |

## 3. 데이터 암호화

### SSE (Server-Side Encryption) 종류
- **SSE-S3:** AWS 관리형 키. 무료이며 가장 일반적인 데이터에 사용. 키 사용 이력을 별도로 확인할 수 없음(추적 불가)
- **SSE-KMS:** KMS 관리형 키. Audit Log(키 사용 이력)가 남아 추적 가능. 키 별로 사용 권한을 제어할 수 있어 컴플라이언스 준수에 유리.
- **SSE-C:** 고객 제공 키. 키를 AWS에 저장하지 않는 극비 데이터용.

### 암호화 표준 방식
- **표준 방식:** **전송 구간 암호화(HTTPS)** + **저장 구간 암호화(SSE-S3/KMS)** 조합 사용.
- **cf. 클라이언트 측 암호화(Client-Side Encryption)** 
    - 서버가 데이터 내용을 볼 수 없어 가공(ex. 썸네일 생성)이 불가능
        - 데이터를 보낸 사람과 받는 사람 외에는 서버(AWS)조차 내용을 절대 알면 안되는(종단간 암호화) 특수한 경우에만 제한적으로 사용(ex. 텔레그램 비밀대화)
