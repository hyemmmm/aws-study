# 오브젝트 스토리지(Object Storage)와 객체(Object)란 무엇인가?

오브젝트 스토리지는 대량의 비정형 데이터(이미지, 영상, 로그 등)를  
**객체(Object)** 단위로 저장하고 관리하는 스토리지 방식이다.

파일이나 폴더 구조가 아닌 **평면(Flat) 구조**로 데이터를 저장하며,  
각 객체는 데이터 자체와 풍부한 메타데이터, 그리고 고유 식별자를 가진다.

대표적인 예로 Amazon S3가 있다.

![Object Storage](https://miro.medium.com/v2/resize:fit:1120/1*Qb0YNyBnozctT5kIxwu36w.png)

## 1. 객체(Object)의 정의

오브젝트 스토리지에서 말하는 **객체(Object)** 는 다음 세 요소로 구성된다.

**Object = Data + Metadata + Unique Identifier**

### 데이터 (Data)
- 실제 저장되는 내용
- 예:
  - 이미지, 영상, 오디오
  - 로그 파일
  - JSON, CSV, 바이너리 데이터

### 메타데이터 (Metadata)
- 데이터에 대한 설명 정보
- 예:
  - Content-Type (`image/png`)
  - 생성 시각
  - 암호화 여부
  - 사용자 정의 메타데이터
- 파일 시스템보다 **훨씬 풍부하고 자유롭게 확장 가능**

### 고유 식별자 (Object ID)
- 객체를 식별하는 유일한 값
- S3 기준으로는 다음 조합으로 식별됨

Bucket + Key

예시:  
my-bucket/images/2026/01/cat.png

이 세 요소가 하나로 묶인 독립적인 저장 단위가 **객체(Object)** 다.

---

## 2. 오브젝트 스토리지의 핵심 개념

### 객체 기반(Object-based)
- 파일을 단순한 파일이 아닌 **객체**로 취급
- 각 객체는 데이터, 메타데이터, 고유 ID를 포함

### 평면 구조(Flat Namespace)
- 전통적인 디렉토리/폴더 구조가 없음
- Key의 prefix를 통해 디렉토리처럼 *보이게* 할 뿐, 실제 계층 구조는 아님
- 구조적 제약이 없어 **확장성이 매우 뛰어남**

### API 기반 접근
- POSIX 파일 시스템이 아닌 **REST API 기반**
- HTTP 방식으로 객체에 접근
  - PUT: 객체 업로드
  - GET: 객체 조회
  - DELETE: 객체 삭제

---

## 3. 파일 시스템과의 비교

| 구분 | 파일 시스템 | 오브젝트 스토리지 |
|---|---|---|
| 저장 단위 | 파일 | 객체 |
| 구조 | 계층적 디렉토리 | 평면 구조 |
| 식별 방식 | 경로 기반 | 고유 ID(Key) |
| 수정 방식 | 부분 수정 가능 | 전체 객체 재업로드 |
| 메타데이터 | 제한적 | 풍부하고 확장 가능 |
| 접근 방식 | POSIX | REST API |

---

## 4. 장점

- 대용량 비정형 데이터 관리에 적합
- 높은 확장성과 비용 효율성
- 메타데이터 기반 관리 용이

---

## 5. 단점 및 한계

- 객체 수정 시 전체 재업로드 필요
- 실시간 트랜잭션 처리에는 부적합

---

## 6. 주요 사용 사례

- 데이터 레이크
- 백업 및 아카이빙
- 미디어 저장
- 웹/모바일 애플리케이션 콘텐츠 저장
