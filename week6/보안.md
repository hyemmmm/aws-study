**\[1\] AWS 보안 이해하기**

## **1.1 정보 보안의 3대 요소** 

정보 보안의 핵심은 단순히 '잠그는 것'이 아니라, 다음 세 가지 균형을 맞추는 것이다.

* **기밀성 (Confidentiality):** 허락된 사람만 허용 (암호화, 접근 제어)  
* **무결성 (Integrity):** 데이터가 변하지 않았음을 증명. (디지털 서명, 변경 이력 관리)  
* **가용성 (Availability):** 필요할 때 언제든 쓸 수 있어야 함. (이중화, 백업, 복구 체계)

**1.2 온프레미스 vs. 클라우드 보안의 차이**  
물리적 공간을 직접 통제하느냐, 논리적인 권한 설정을 통해 통제하느냐의 차이이다.

| 구분 | 온프레미스 (On-premise) | AWS (Cloud) |
| :---- | :---- | :---- |
| **물리 보안** | 데이터 센터 출입 통제, 랙(Rack) 잠금 | **AWS가 책임짐** (사용자는 신경 안 써도 됨) |
| **관리 주체** | 사용자가 하드웨어부터 소프트웨어까지 전체 관리 | **공동 책임 모델** (인프라는 AWS가, 데이터와 설정은 사용자가) |
| **위험 요소** | 물리적 장비 고전, 도난 등 | **계정 탈취**, 잘못된 설정으로 인한 리소스 오남용 |

**1.3 AWS 보안의 핵심: IAM과 Root 계정 관리**  
클라우드에서는 '계정 권한'이 곧 온프레미스의 '서버실 열쇠'와 같다.

### **① Root 계정은 '마스터키'입니다**

* Root 계정은 모든 권한을 가진 절대 권한자.  
* **위험성:** 탈취 시 해커가 고사양 EC2 인스턴스를 대량 생성하여 가상화폐 채굴에 이용(빌링 폭탄)하거나, 데이터를 삭제할 수 있다.  
* **대책:** 가급적 사용을 자제하고, **2FA(MFA, 다요소 인증)** 설정이 필수.

### **② IAM (Identity and Access Management)**

사용자별로 필요한 최소한의 권한만 부여하는 것이 원칙.

* **인증(Authentication):** 누구인지 검증 (로그인 아이디/비번, MFA)  
* **인가(Authorization):** 가능한 영역 검증 (특정 S3 버킷 읽기 권한 등)  
* **최소 권한의 원칙:** 업무에 꼭 필요한 권한만 할당하여 사고 발생 시 피해 범위를 최소화해야 한다..

**Q1\. Root 계정 대신 IAM 사용자를 쓰라는 건 알겠는데, 내 소스 코드(APP)는 어떻게 권한을 얻나?**  
가장 위험한 실수 중 하나가 **IAM 사용자의 Access Key와 Secret Key를 코드나 설정 파일에 하드코딩**하는 것.

* **해결책: IAM Role(역할) 사용**  
  * 특정 권한이 담긴 **IAM Role**을 생성하고, 이를 EC2 인스턴스나 Lambda 함수에 해당 역할을 '부여'한다. 열쇠대신 이름표를 시스템에 사용하는 것과 비슷하다.  
  * 그러면 코드 내에서 별도의 키 입력 없이도 AWS SDK가 자동으로 임시 토큰을 발급받아 보안을 유지한다. 깃허브에 키가 유출될 걱정을 원천 차단하는 방법이다.

**Q2\. 보안 그룹(Security Group)과 네트워크 ACL(NACL)의 차이는 뭔가?**  
방화벽 설정 시 자주 혼동하는 부분. "왜 설정했는데 접속이 안 되지?"라는 상황에서 꼭 체크해야 한다.

* **보안 그룹 (Security Group):** **인스턴스 단위**의 방화벽. '허용' 규칙만 설정 가능하며, 들어온 요청은 나갈 때 자동으로 허용되는 **Stateful** 방식입니다.   
* **네트워크 ACL (NACL):** **서브넷 단위**의 방화벽. '거부' 규칙도 설정 가능하며, 들어온 요청이라도 나가는 규칙을 따로 설정해야 하는 **Stateless** 방식.

| 구분 | 보안 그룹 (Security Group) | 네트워크 ACL (NACL) |
| :---- | :---- | :---- |
| **적용 범위** | 인스턴스 (EC2 등) | 서브넷 (Subnet) |
| **상태 유지** | **Stateful** (응답 자동 허용) | **Stateless** (응답 규칙 필요) |
| **우선순위** | 모든 규칙 검사 후 허용 | 규칙 번호 순서대로 적용 |

**Q3\. 실수로 권한을 너무 많이 줬을 때 발생하는 문제는?**  
이는 '최소 권한의 원칙'에 위배됨.

* **위험성:** 만약 웹 서비스에 취약점이 있어 서버가 해킹당했다면, 해커는 서버에 부여된 권한을 그대로 이어받는다. 이때 권한이 너무 크면 해커가 DB를 삭제하거나 다른 계정의 정보를 탈취하는 등 피해 범위가 걷잡을 수 없이 커진다.  
* **해결책:** **IAM Policy Simulator**를 활용한다. 내가 작성한 정책이 정확히 어떤 액션만 허용하는지 미리 테스트해 볼 수 있다.

**\[2\] AWS를 안전하게 사용하기 위한 IAM** 

## **2.1 AWS Identity and Access Management란** 

IAM은 AWS 리소스에 **'누가(인증)', '무엇을(인가)'** 할 수 있는지 통제하는 관문.

* **핵심:** 단순히 로그인 아이디를 만드는 곳이 아니라, 사람과 서비스(프로그램) 모두의 권한을 중앙에서 제어하는 컨트롤 타워이다.  
* **특징:** 모든 AWS 조작(EC2 생성, S3 조회 등)은 반드시 IAM의 허락을 거쳐야만 실행된다.

**2.2  IAM 사용자, IAM 그룹**   
실제 AWS를 사용하는 '사람'이나 '개별 실체'를 관리하는 단위.

* **IAM 사용자:** 이름과 비밀번호(콘솔용) 또는 액세스 키(CLI/프로그램용)를 가진 개별 계정.  
* **IAM 그룹:** 사용자들의 공통 집합. (예: 개발팀, 운영팀)  
  * **장점:** 개별 사용자마다 일일이 권한을 주는 대신, 그룹에 권한을 주고 사용자를 넣었다 빼기만 하면 되므로 관리가 매우 편리하다.

**2.3  IAM 정책**   
"무엇을 할 수 있는가"를 정의한 권한 설명서.

* **AWS 관리형 정책:** AWS가 미리 만들어둔 정책. (예: ReadOnlyAccess) 서비스가 업데이트되면 AWS가 알아서 권한도 업데이트해 준다.  
* **고객 관리형 정책:** 보안을 위해 딱 필요한 권한만 세밀하게 직접 만든 정책.  
* **유연성:** 하나의 정책을 여러 사용자나 그룹, 역할에 중복해서 붙일 수 있다.

**2.4 IAM 역할**   
특정인에게 고정된 권한을 주는 것이 아니라, 필요할 때만 빌려 쓰는 임시 권한'.

* **안전모 비유:** 평소에는 권한이 없다가, 작업할 때만 '안전모(Role)'를 써서 임시로 권한을 얻는 방식.  
* **보안성:** 유출 위험이 큰 '고정된 액세스 키' 대신, 자동으로 사라지는 '임시 보안 자격 증명'을 사용하므로 훨씬 안전하다. 서비스 간(예: Lambda가 EC2를 조작할 때) 권한 부여에 필수적이다.

**2.5 AWS 계정 분리 및 Role Switch**   
보안과 관리를 위해 계정을 여러 채로 나누어 사용하는 전략.

* **계정 분리:** 개발(Dev), 테스트(Test), 운영(Prod) 환경을 아예 다른 계정으로 분리하여 사고 발생 시 피해가 다른 환경으로 퍼지는 것을 막는다.  
* **Role Switch:** 여러 계정마다 사용자를 일일이 만들면 관리가 힘들다. 메인 계정에만 사용자를 만들고, 다른 계정에는 '역할'만 둔 뒤 필요할 때만 '역할 전환(Switch)'을 통해 로그인 없이 넘나들며 작업하는 효율적인 방식이다.

 **Q1\. 정책(Policy)을 직접 짤 때, JSON 코드가 너무 복잡하다. 쉽게 짜는 법 없나?**

**IAM 정책은 JSON이라는 형식으로 작성되는데, 처음 보면 { "Effect": "Allow", "Action": ... } 같은 구조가 낯설고 어렵다.**

* 해결책: AWS 정책 생성기(Visual Editor) 활용  
  * 메모장에 코드를 직접 칠 필요가 없다. AWS 콘솔의 '시각적 편집기'를 사용하면 드롭다운 메뉴에서 서비스(예: S3)와 동작(예: ListBucket)을 선택하기만 해도 JSON 코드가 자동으로 생성된다. 
* 답변: 처음부터 JSON을 직접 쓰지 말고 관리형 정책을 복사해서 수정하거나 시각적 편집기를 통해 구조를 익히는 것이 오타로 인한 보안 틈을 줄이는 지름길이다.

**Q2\. 로컬 PC에서 개발할 때는 '역할(Role)'을 부여할 수 없는데, 이땐 키를 써야 하나?**  
앞서 EC2나 Lambda에는 '역할(Role)'을 준다고 배웠지만, 로컬 환경은 AWS 서비스가 아니기 때문에 '역할'을 직접 걸어줄 수 없다.

* **해결책: 환경 변수와 AWS CLI Profile 활용**  
  * 코드 안에 키를 박는 대신, 내 컴퓨터의 환경 변수에 키를 저장하거나 \~/.aws/credentials 파일에 프로필을 만들어 관리한다.  
* 답변: "로컬에서는 액세스 키를 쓰되, 절대 코드에 넣지 않는다. 대신 aws configure 명령어를 통해 PC 자체에 저장하고, 코드가 이를 찾아 읽게 만들어야 한다. Git에 올릴 때는 .gitignore 파일로 이 설정 파일이 올라가지 않게 막는 것이 필수다.

**Q3\. Role Switch는 로그아웃하고 다시 로그인하는 거랑 뭐가 다른가?**  
계정이 여러 개(개발용, 운영용)일 때 매번 로그아웃하고 다시 로그인하는 건 매우 번거롭고 시간이 아깝다.

* **해결책:**   
  * Role Switch 기능을 쓰면, 'A 계정의 나'가 로그아웃 없이 'B 계정의 관리자'로 순식간에 변신할 수 있다.  
* 답변: 브라우저 탭을 여러 개 띄우거나 브라우저를 껐다 켤 필요가 없다. 콘솔 우측 상단의 이름표를 눌러 'Switch Role'만 클릭하면 된다. 이렇게 하면 실수로 운영 계정에서 작업하는 사고도 줄일 수 있고, 비밀번호 관리도 메인 계정 하나만 하면 되니 훨씬 효율적이다.

**\[3\] 계정 내 작업 이력 기록 \- AWS CloudTrail**

CloudTrail은 AWS 계정에서 발생하는 "누가, 언제, 어디서, 무엇을 했는가"에 대한 모든 발자취를 기록하는 서비스.

\* 감시와 조사 : 사용자의 실수나 해킹 같은 악의적인 조작을 사후에 추적하고 증명하는 중요한 근거가 된다.  
\* 기본 저장 : 기본적으로 90일간 기록이 자동 저장되며, 더 오래 보관하려면 S3 등에 별도로 저장(트레일 생성)해야 한다.

**3.1 기록되는 내용**

CloudTrail은 AWS API를 통한 조작을 기록한다. 단, 서버(EC2) 내부에서 직접 명령어를 치는 것까지는 알 수 없다.

\* 관리 이벤트 (Management Events) : 로그인, 리소스 생성/삭제 등 인프라의 '틀'을 바꾸는 행위. (90일 로그 무료)  
\* 데이터 이벤트 (Data Events) : S3의 파일을 읽거나 Lambda를 실행하는 등 실제 '데이터'를 다루는 행위. (별도 설정 및 유료)  
\* 인사이트 이벤트 (Insight Events) : 평소와 다른 비정상적인 행동이 감지되었을 때만 특별히 기록. (유료)

**3.2 트레일 출력**

기록된 로그는 영구 보관이나 실시간 알림을 위해 다른 서비스로 보낼 수 있다.

\* S3 (스토리지) 출력 :  파일 형태로 저장되며, 무기한 보관이 가능하다.  
\* 무결성 검사 : 저장된 로그 파일이 누군가에 의해 수정되거나 조작되지 않았음을 증명하는 기능이 포함된다.

\* CloudWatch Logs 출력 : 로그를 '스트림(실시간 흐름)' 방식으로 전송.  
\* 실시간 감시 : 특정 위험한 작업(예: Root 계정 로그인)이 발생했을 때 즉시 알림을 보내도록 설정할 수 있어 대응 속도가 빠르다.

**Q: 서버(EC2) 안에서 \`rm \-rf /\`를 입력해서 파일을 지웠는데 왜 CloudTrail에 안 나오나?**

A : CloudTrail은 'AWS라는 집의 관리 이력'을 기록하는 것이지, '방 안(서버 내부)의 일기'를 쓰는 게 아닙니다. 서버 내부 작업까지 기록하려면 CloudWatch Agent라는 별도의 프로그램을 설치해야 한다.

**Q: S3에 로그를 쌓으면 돈이 많이 나오지 않나?**

A : 로그 파일 자체는 텍스트라 용량이 작지만, 매일 수만 건의 데이터 이벤트(S3 읽기/쓰기 등)를 다 기록하면 요금이 상당히 나올 수 있다. 그래서 실무에서는 꼭 필요한 중요 이벤트만 골라서 기록하는 설정을 선호한다.

**\[4\] 설정 내역 및 설정 내용을 자동으로 관리 \- AWS Config**

AWS Config는 AWS 리소스가 "과거에 어떤 설정이었고, 지금은 어떻게 변했는가"를 기록하는 서비스.

### **4.1 설정 이력을 저장하는 AWS Config**

* **리소스 중심 기록 :** CloudTrail이 '행동(누가 했나)'을 기록한다면, Config는 '결과(어떻게 변했나)'를 기록. (예: 보안 그룹의 80번 포트가 열린 상태 그 자체를 기록)  
* **이력 관리:** 기본 **7년**간 설정 변화를 저장하여, 사고 발생 시 "사고 직전의 설정 상태"를 즉시 확인할 수 있다.  
* **고급 쿼리:** SQL 같은 문법을 사용해 "현재 내 계정에 있는 모든 EC2 인스턴스 개수" 등을 쉽게 조회할 수 있다.  
* **비용 관리:** 자원 수가 많아지면 요금이 늘어나므로, 중요한 자원 유형만 골라서 기록하는 설정이 권장된다.

### **4.2 설정을 확인하는 AWS Config 규칙**

설정이 보안 원칙에 맞는지 실시간으로 검사하고 고치는 파수꾼 역할을 한다.

* **컴플라이언스 체크:** "S3가 퍼블릭으로 열려 있나?", "DB가 암호화되었나?" 같은 규칙을 정해두고 위반 사항을 찾아낸다.  
* **알림 및 자동 수정:** 규칙 위반(미준수) 발견 시 이메일 알림을 보내거나, **'자동 문제 해결'** 기능을 통해 즉시 설정을 원상복구(예: 열린 S3 버킷을 즉시 닫음)한다.  
* **기술적 구현:** 자동 수정은 **AWS Systems Manager(SSM)의 Automation** 기능을 빌려와 실행한다.

**Q: "자동 문제 해결 기능을 켜두면 오히려 위험하지 않나? 테스트하려고 잠시 열어둔 건데 Config가 강제로 닫아버리면 업무에 방해가 될 가능성도 있을것 같다.**

* **답변:** 실무에서는 모든 규칙에 자동 수정을 걸지 않는다. '운영(Prod) 환경'에서는 엄격하게 자동 수정을 걸어 보안 사고를 원천 차단하지만, '개발(Dev) 환경'에서는 알림만 오게 설정하여 개발의 자유도를 보장하는 식으로 나누어 관리한다.

**Q: Config 규칙을 직접 만들 수도 있나? 예를 들어 특정 태그(Tag)가 없는 리소스는 다 찾아내고 싶을때."**

* **답변:** 가능하다. AWS가 제공하는 **관리형 규칙** 외에도, **Lambda 함수**를 이용해 직접 **사용자 지정 규칙**을 만들 수 있다. "이름표(Tag)가 없는 서버는 1시간 뒤에 삭제하라" 같은 복잡한 비즈니스 로직도 구현할 수 있다.

**\[5\] AWS 작업 내용 감시 \- AWS GuardDuty**

## 

### **5.1 지능형 위협 탐지 서비스 Amazon GuardDuty**

GuardDuty는 AWS 계정 내부에서 일어나는 수상한 움직임을 머신러닝(AI)으로 찾아내는 보안 탐지기이다.

* **즉시 활성화:** 클릭 한 번으로 시작할 수 있으며, 계정 생성 직후 바로 켜두는 것이 권장된다.  
* **탐지 범위:** 계정 도용, 리소스 오남용, 인스턴스 해킹 등을 감시한다.  
  * **예시:** 갑작스러운 Root 계정 사용, 액세스 키 대량 사용(탈취 의심), EC2가 공격용 좀비 PC로 변함 등.  
* **분석 데이터:** CloudTrail(활동), VPC 흐름 로그(네트워크), DNS 로그를 기반으로 분석한다. (단, 서버 내부 앱이나 Lambda 코드 자체의 위협은 탐지 불가)  
* **지능형 학습:** 평소 사용 패턴을 학습하여, "평소와 다른 시간/장소에서의 로그인" 같은 비정상 행위를 잡아낸다.  
* **샘플 이벤트:** GuardDuty가 실제로 어떻게 작동하는지 미리 보고 싶다면, 제공되는 **'샘플 생성'** 기능을 통해 가짜 탐지 결과를 만들어 설정을 테스트할 수 있다.

### 

**Q: GuardDuty가 위험을 발견하면 자동으로 차단해주나?**

* **답변:** GuardDuty는 기본적으로 **'탐지(보고)'** 서비스이지 '방어(차단)' 서비스가 아니다. 하지만 **Amazon EventBridge**와 **Lambda**를 연결하면, GuardDuty가 경고를 울리는 즉시 자동으로 서버를 끄거나 계정을 잠그는 '자동 방어 시스템'을 직접 구축할 수 있다.

**Q: 로그 분석을 하면 네트워크 속도가 느려지거나 서버 성능에 영향이 있나?**

* **답변:** 전혀 없다\! GuardDuty는 AWS 내부 인프라에서 발생하는 로그 사본을 가져다 분석하기 때문에, 여러분이 운영 중인 서비스의 성능이나 네트워크 속도에는 **0.1%의 영향도 주지 않는다.**

**Q: 알림이 너무 많이 오면 어떡하나? 중요하지 않은 건 빼고 싶을땐?**

* **답변:** GuardDuty는 탐지 결과에 **'심각도(Severity)' 점수**를 매겨준다. (0.1\~8.9점) 실무에서는 이 점수가 높은(예: 7.0 이상) 긴급한 상황에 대해서만 휴대폰 문자나 Slack으로 알림이 오도록 설정하여 '알림 피로도'를 조절한다.

**\[6\] 웹 응용 프로그램 보안 강화 \- AWS WAF**

네트워크 방화벽(보안 그룹 등)이 "어느 길(IP/Port)로 오는가"를 검사한다면,WAF는 "편지 내용(HTTP 요청 데이터)에 문제가 있는가"를 검사한다.

**6.1 간단하게 적용할 수 있는 WAF**

WAF는 웹 사이트를 노리는 정교한 공격을 클릭 몇 번으로 막아주는 관리형 서비스이다.

\* 적용 대상 : ALB(Application Load Balancer), CloudFront(CDN), API Gateway, AppSync 등 웹 서비스의 입구 역할을 하는 서비스에 바로 연결할 수 있다.  
\* 주요 특징 : 

- 웹 ACL(Access Control List) : 어떤 요청을 허용하고 거부할지 결정하는 규칙 목록이다.  
- 정교한 탐지 : 단순히 IP를 막는 수준을 넘어, SQL 인젝션이나 크로스 사이트 스크립팅(XSS) 같은 복잡한 웹 공격 코드를 분석해서 차단한다.

\* 편리한 규칙 설정 :

- AWS 관리형 규칙 : AWS 보안 전문가들이 미리 만들어둔 '공격 방어 세트'를 그대로 가져다 쓸 수 있다.  
- 마켓플레이스 활용 : 타사(보안 전문 기업)에서 만든 강력한 규칙도 구매하여 적용 가능하다.

\* 비용 효율 : 서버를 직접 구축할 필요 없이, 설정한 규칙 수와 실제 처리된 요청량만큼만 요금을 내면 되므로 경제적이다.

**Q : WAF를 쓰면 웹사이트 속도가 느려지지 않나?**

\* 답변 : 모든 요청을 검사하기 때문에 아주 미세한 지연은 발생할 수 있다. 하지만 AWS 인프라 위에서 동작하므로 사용자가 체감하기는 매우 어렵다. 오히려 해커의 공격 트래픽을 미리 걸러내어 서버의 부하를 줄여주는 긍정적인 효과가 더 크다.

**Q : 정상적으로 보내는 요청도 WAF가 공격으로 오해해서 막으면 어떡하지?**

\* 답변: 처음부터 차단(Block) 모드로 켜지 않고, 개수 산정(Count) 모드로 먼저 실행한다. 로그를 보면서 정상 요청이 걸러지는지 확인한 뒤, 문제가 없을 때 차단 모드로 전환하는 것이 정석이다.

**Q: AWS 관리형 규칙만 써도 충분한가?**

\*  답변 : 웬만한 일반적인 공격은 'AWS Managed Rules'만으로도 90% 이상 막을 수 있다. 하지만 특정 국가의 IP를 차단하거나, 우리 서비스만의 특수한 패턴을 막아야 할 때는 직접 규칙을 추가하는 것이 좋다.

**\[7\] 시스템 보안을 강화하는 6가지 서비스 \- 기타 보안, 인증 서비스** 

### **7.1 AWS Network Firewall** 

VPC(가상 네트워크) 전체를 지키는 '관리형 방화벽'이다.

* **기능:** 단순히 IP나 포트를 막는 것을 넘어, 특정 도메인 접속을 차단하거나 복잡한 침입 방지(IPS) 규칙을 적용할 수 있다.  
* **특징:** 트래픽 양에 따라 자동으로 크기가 조절되며, 보안 그룹보다 훨씬 정교한 통제가 가능하다.

### **7.2 Amazon Inspector**

EC2 인스턴스 내부에 숨은 '취약점 점검관'이다.

* **기능:** 서버에 설치된 소프트웨어의 버전이 너무 낮아 해킹 위험은 없는지, 보안 설정이 AWS 권장 사항을 잘 따르는지 검사한다.  
* **자동화:** **AWS Systems Manager**와 연결하면 취약점이 발견된 서버에 보안 패치를 자동으로 설치할 수도 있다.

### **7.3 AWS Shield**

DDoS 공격(대량의 가짜 트래픽으로 서버를 마비시키는 공격)을 막는 '방패'이다.

* **Standard (무료):** 모든 AWS 사용자에게 자동으로 적용되며, 일반적인 공격을 알아서 막아준다.  
* **Advanced (유료):** 아주 강력하고 집요한 공격에 대응하며, AWS 보안 전문가의 1:1 지원을 받을 수 있다. (비용이 매우 비싸 대규모 서비스용입니다.)

### **7.4 AWS Security Hub**

내 계정의 보안 상태를 한눈에 보여주는 '종합 점검 대시보드'이다.

* **기능:** GuardDuty, Inspector, Config 등 여러 서비스의 보안 경고를 한곳에 모아 보여준다.  
* **기준:** 국제적인 보안 표준(PCI DSS, CIS 등)을 우리 시스템이 얼마나 잘 지키고 있는지 점수표로 보여주어 보안 사각지대를 없애준다.

### **7.5 AWS Cognito**

앱 사용자를 위한 '회원가입 및 로그인 관리자'이다.

* **기능:** 일반 사용자가 우리 앱에 가입하고 로그인하는 기능을 직접 코딩할 필요 없이 빌려 쓸 수 있다.  
* **연동:** 구글, 페이스북 같은 소셜 로그인이나 다요소 인증(MFA) 기능을 쉽게 추가할 수 있다.

### **7.6 AWS Directory Service**

회사 내의 '사용자 계정 및 장치 관리 명부'이다.

* **기능:** 주로 기업에서 사용하는 Microsoft Active Directory(AD)를 클라우드에서 구현해 준다.  
* **활용:** 직원들이 회사 컴퓨터에 로그인하는 아이디로 AWS 서비스나 윈도우 서버에도 똑같이 로그인할 수 있게 통합해 준다.

**Q: AWS Shield Standard만 믿어도 되나? 해커가 공격해서 요금이 폭탄처럼 나오면 어떻게 해야 하나?**

**답변:** Standard는 일반적인 공격은 잘 막아주지만, 만약 공격 때문에 예상치 못한 트래픽 요금이 발생했다면 **Shield Advanced** 가입자의 경우에만 해당 요금을 환불(DDoS 요금 보호)해 주는 혜택이 있다. 일반 사용자라면 **AWS Budgets**를 설정해 요금이 급증할 때 즉시 알림을 받도록 하는 것이 최선이다.

**Q: Cognito를 쓰면 제가 회원 정보를 직접 DB에 저장 안 해도 되나?**

* **답변:** 네\! 사용자의 아이디, 암호화된 비밀번호, 이메일 등을 Cognito가 안전하게 관리해 준다. 개발자는 DB 보안 걱정을 덜 수 있고, 사용자가 비밀번호를 잊어버렸을 때 찾는 기능 등도 자동으로 구현되니 아주 편리하다
